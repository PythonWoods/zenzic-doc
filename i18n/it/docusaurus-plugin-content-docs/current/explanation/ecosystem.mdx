# Il Modello Adapter

Zenzic opera come **revisore esterno sovrano**. Non si incorpora mai all'interno di un motore di build,
non avvia mai sottoprocessi e non richiede mai extra di installazione specifici per il motore.
Un unico binario. Ogni motore. Protezione completa.

---

## Adapter â€” Il Layer dell'Intelligenza {#adapters}

Un **Adapter** (in `zenzic.core.adapters`) Ă¨ un componente Pure-Python che **legge** il file di
configurazione di un motore di build e lo traduce nel modello interno di Zenzic. Gli adapter rispondono
a tre domande:

1. **Quali file sono navigabili?** (`get_nav_paths()`) â€” Quali file `.md` / `.mdx` compaiono nella navigazione del sito?
2. **Quale URL ottiene questo file?** (`get_route_info()`) â€” URL canonico, slug override, route base path.
3. **Quali pattern ignora questo motore?** (`get_ignored_patterns()`) â€” File come `README.md` che alcuni motori saltano.

Gli adapter sono **stateless e senza sottoprocessi**. Un `MkDocsAdapter` legge `mkdocs.yml` in modo

MkDocs puĂ˛, con un adapter diverso, analizzare un progetto Docusaurus o Zensical usando un percorso
di codice identico.

## Adapter Integrati {#built-in-adapters}

| Classe adapter | Motore | File di configurazione letto |
| :--- | :--- | :--- |
| `MkDocsAdapter` | `mkdocs` | `mkdocs.yml` |
| `ZensicalAdapter` | `zensical` | `zensical.toml` |
| `DocusaurusAdapter` | `docusaurus` | `docusaurus.config.js` / `.ts` |
| `StandaloneAdapter` | `standalone` | _(nessuno â€” ogni file Ă¨ REACHABLE)_ |

Gli adapter vengono scoperti tramite il gruppo entry-point `zenzic.adapters`. Puoi pubblicare un
adapter di terze parti per qualsiasi motore senza toccare il core di Zenzic:

```toml
# pyproject.toml del tuo adapter
[project.entry-points."zenzic.adapters"]
mio_motore = "mio_pacchetto.adapter:MioAdapter"
```

## Strategia CLI Sovrana {#sovereign-cli}

Zenzic v0.7.0 non fornisce plugin per motori di build nĂ© integrazioni interne.
Questa Ă¨ una decisione architetturale deliberata, non una funzionalitĂ  mancante.

### PerchĂ© nessun plugin per motori?

| Aspetto | Plugin nel motore | CLI Sovrana |
| :--- | :--- | :--- |
| Scansione credenziali (Shield) | âťŚ Strutturalmente assente â€” gli hook si attivano troppo tardi | âś… Hardening completo ZRT-006/007 |
| Path traversal (Blood Sentinel) | âťŚ Il motore controlla la risoluzione dei path | âś… Ogni link normalizzato da Zenzic |
| Validazione dei link | âťŚ Resolver del motore â€” non la VSM | âś… VSM O(1) con cache SHA256 |
| ParitĂ  tra motori | âťŚ Plugin MkDocs â‰  Plugin Docusaurus | âś… Percorso di codice unico per tutti i motori |
| Isolamento dipendenze | âťŚ Richiede `pip install zenzic[motore]` | âś… `pip install zenzic` â€” fatto |

Un plugin che vive all'interno di un motore di build non puĂ˛ garantire le stesse proprietĂ  di un
revisore esterno. Lo Shield, il Blood Sentinel e la VSM sono stati progettati per l'esecuzione
sovrana. Incorporarli in un hook di build li degraderebbe a controlli best-effort.

### Il workflow consigliato

```bash
# Installazione â€” un solo pacchetto, nessun extra
pip install zenzic

# Rilevamento motore dai file di configurazione, scrittura zenzic.toml
zenzic init

# Audit completo â€” VSM + Shield + Blood Sentinel + validazione link
zenzic check all

# In CI (GitHub Actions, GitLab CI, ecc.)
# Aggiungi come step prima o dopo la build, non al suo interno
zenzic check all --strict
```

Zenzic viene eseguito **prima o dopo** la build â€” mai al suo interno. Questo garantisce:
- Exit 2 per qualsiasi credenziale esposta (la shell si ferma immediatamente)
- Exit 3 per qualsiasi link path-traversal (non sopprimibile)
- Risultati engine-agnostici: lo stesso comando `zenzic check all` funziona per ogni motore

## Scegliere il Modello Giusto {#choosing}

| Scenario | Approccio consigliato |
| :--- | :--- |
| Pipeline CI per qualsiasi motore | `zenzic check all` â€” aggiungi uno step, nessun plugin necessario |
| Gate pre-commit per le credenziali | `zenzic check references` â€” si registra come hook pre-commit |
| Motore custom non ancora supportato | **Scrivi un Adapter** â€” pubblica come pacchetto separato, registra via entry-point `zenzic.adapters` |
| Migrazione da MkDocs a Docusaurus | Usa `zenzic check all` con `engine = "mkdocs"` sulla sorgente, `engine = "docusaurus"` sulla destinazione |
